## TL;DR

The monitoring PRs (#22, #23, #24) merged to `main` but never reached any cluster because the live ArgoCD `Application` CRs have drifted from the YAMLs in `argo/`. Once that's fixed, three pre-existing prod problems become visible: `echo-agent` has been broken for ~4 weeks, one node is at 97% CPU and starving DaemonSets, and a couple of recurring log patterns deserve tickets.

## P0 — Blocks shipping monitoring

### 1. `echo-monitoring-prod` Application is tracking the wrong branch

| | |
|---|---|
| Repo (`argo/echo-monitoring-prod.yaml`) | `targetRevision: main` (HEAD `2ee97c4`) |
| Live CR on prod cluster | `targetRevision: feature/monitoring_prod` (pinned at `f6bf537`, 2025-09-04) |
| Last reconcile | 2026-05-14 13:04 UTC (ArgoCD is polling — just polling the wrong branch) |

**Why it happened.** There's no app-of-apps. Editing the Application YAML in git does nothing until someone re-runs `kubectl apply -f argo/echo-monitoring-prod.yaml`. The current live CR was applied months ago against a feature branch and never updated.

**Evidence of impact.**
- `alertmanager-config` ConfigMap age: 274 days. Has only one receiver (`slack-prod`), no `blackhole` route, no severity split → PR #23 not deployed.
- No CronJobs in the `monitoring` namespace → PR #24 (weekly warning digest) not deployed.
- `helm/monitoring/templates/...` regex change from PR #22 not in rendered config.

**Fix.** From prod kube-context:
```
kubectl apply -f argo/echo-monitoring-prod.yaml
```
ArgoCD's next refresh (≤3 min) will then sync `monitoring` namespace to `2ee97c4`.

### 2. `echo-monitoring-dev` Application is not deployed at all

| | |
|---|---|
| Repo (`argo/echo-monitoring-dev.yaml`) | exists, valid |
| Live CRs on dev cluster | only `echo-dev` — no `echo-monitoring-dev` |
| `monitoring` namespace on dev | does not exist; no Prometheus/Grafana/Loki/Alertmanager |

The dev cluster has had zero observability stack since it was provisioned.

**Fix.** From dev kube-context:


The same gap exists on `dbr-echo-testing-k8s-cluster` (no monitoring Application). Decide separately whether testing needs its own observability or can rely on prod's.

## P1 — Live prod degradation, unrelated to monitoring

### 3. `echo-agent` in `ImagePullBackOff` for ~29 days (prod)

- Live Deployment desires image: `dbr-echo-agent:62ab30fdc85163cabcc9b13479e06191dc460c64` (v1.17.0).
- Registry state for that tag: manifest entry exists with **0 bytes** of layers — blobs gone (GC'd or never pushed).
- `helm/echo/values-prod.yaml` `global.imageTag` is now `a4b23c5d…` (v1.17.2) — also 0 bytes in the registry.
- Latest layer-backed tag in `dbr-echo-agent`: `d22d401` (2026-04-23).
- ArgoCD says `echo-prod` is `OutOfSync` + `Degraded`: it knows desired ≠ live, just isn't applying.

**Open question:** is `agent.enabled: true` intentional in prod right now? If the build pipeline isn't reliably pushing this image, either fix the pipeline or set `agent.enabled: false` in `values-prod.yaml` until it is. The new alertmanager severity rules (once #1 is fixed) will start surfacing this in `#prod-alerts`.

### 4. Node `default-pool-6c3vu` is at 97% CPU / 92% memory requests (prod)

Hosts: argocd, cert-manager, ingress-nginx, coredns, echo-api, echo-directus, echo-worker, csi-do-node, do-node-agent, konnectivity-agent.

Consequence: rolling updates for `node-exporter` and `promtail` DaemonSets have one pending pod each for >25h — that node has no metrics or logs being collected, and the monitoring app is permanently in `Health: Progressing`.

**Fix paths (pick one):**
- Upsize the default-pool node spec.
- Add a node to the pool and let the scheduler rebalance.
- Move ingress-nginx or one of the argocd controllers to `monitoring-devops-pool` via nodeSelector.
- Reduce a request (echo-directus is at 21d age — easy candidate to look at).

## P2 — Recurring log patterns (file as tickets, not on-call)

### Prod (`echo-prod` namespace, last 1h)

| Count | Pattern | Action |
|---|---|---|
| 16 | `asyncio: Task was destroyed but it is pending!` | API code is not awaiting/cancelling background tasks on shutdown. Code-side fix. |
| 16 | `LiteLLM Router: failed to count tokens. Invalid content type: <class 'dict'>. Expected str or dict.` | A call site is passing a dict where LiteLLM wants a string or list-of-strings. Router falls back to default count → requests still succeed, but routing decisions are based on bad data. |
| 2 | `asyncio: Unclosed client session` | aiohttp client session leak — same async-cleanup family as the first row. |
| 1 chain | Transcription chunk arrives with empty `conversation_id` → `Invalid foreign key ""` in `processing_status` collection → cascades through `dramatiq.worker` and `task_transcribe_chunk`. | Producer-side bug enqueueing chunks without a conversation id. One occurrence per hour, but the cascade is noisy. |

| Count | Pattern | Action |
|---|---|---|

## P3 — Cosmetic / informational

- **Prometheus pod graveyard.** ~50 old `Error`/`OOMKilled`/`ContainerStatusUnknown` pods in `monitoring` namespace. Current pod has 0 restarts since 2026-05-13. K8s default GC hasn't reaped them. Cleanup: `kubectl -n monitoring delete pod -l app=prometheus --field-selector=status.phase!=Running`.
- **`echo-k6-dev` / `echo-k6-prod` Applications** show `OutOfSync` + `Missing` on prod cluster. k6 is on-demand load testing per `scripts/k6/README.md` — likely intentional. Confirm and either remove the Applications or document the "Missing is expected" status.

## Recommended order

1. **Apply both monitoring Applications** (P0 #1, #2). Safe and reversible. Dev first, verify sync to `2ee97c4`, then prod.
2. **Verify** PRs #22/#23/#24 land:
   - `kubectl -n monitoring get cronjob` shows the weekly digest cronjob.
   - `kubectl -n monitoring get cm alertmanager-config -o jsonpath='{.data}'` contains `blackhole` receiver and severity-based routing.
   - Grafana panel regex change is visible in the dashboard JSON.
3. **Then** triage P1 #3 (`echo-agent` image) and P1 #4 (node CPU). Not blockers, but the new alertmanager rules will start surfacing them once they're live.
4. P2 log patterns → tickets, not on-call work.
